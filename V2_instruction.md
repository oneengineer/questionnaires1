# 相亲向多维度画像 & 问卷维度指南 v2

> 用途：为「以找对象为目的的社交网站」设计问卷时，提供**维度层面的地图**。  
> 本版本是在前一版通用人格维度的基础上，结合本项目规则进行筛选与补充，**专门面向相亲/婚恋场景**。

---

## 0. 标记说明：什么要重点做、什么已经有了、什么可以放弃

在下面所有维度中，会用三个标记：

- ✅ **核心维度**：  
  需要重点出题，和找对象高度相关，且目前**尚未被现有问卷充分覆盖**。
- 🟡 **已覆盖维度（请勿重复造题）**：  
  已在现有问卷中有较多题目，只作为「画像维度索引」，**AI 不要再生成同类/高度相似问题**。
- ⚪ **可弱化 / 谨慎使用维度**：  
  对匹配度帮助有限、或太抽象/太学术，仅在有明确产品需求时再展开。

---

## 1. 用户基本背景维度（自我介绍层）

### 1.1 基本人口学信息（自报）

- 年龄、性别、身高、体重区间（是否展示可按产品策略控制）。  
- 城市 / 所在地区、是否愿意异地 / 跨城发展关系。  
- 学历、职业类型、工作行业、工作时间规律（朝九晚五、轮班、自由职业等）。  
- 当前收入区间（可粗分档）。  

> 标记：✅ 核心维度  
> 说明：这些是相亲场景的最低配背景信息，也是后续分支逻辑的重要入口。

---

### 1.2 当前生活状态与居住状态

- 独居 / 与室友同住 / 与家人同住 / 宿舍等。  
- 主要生活城市是否稳定（准备长期在此 vs 计划几年内迁移）。  
- 是否有长期照顾责任（如照顾父母、家中病人等）。  

> 标记：✅ 核心维度  
> 说明：和「生活节奏、家庭责任、是否愿意搬家」等后续分支强相关。

---

### 1.3 原生家庭氛围（**轻量版**）

- 对父母（或监护人）关系的总体感受：  
  - 多数时间和谐 / 经常冷战 / 经常争吵 / 比较疏远等。  
- 自己和父母的相处方式感受：  
  - 支持型 / 控制型 / 忽视感 / 高期待高压力等（用行为化描述）。  

> 标记：⚪ 可选维度（轻量）  
> 说明：与亲密关系模式有关，但偏隐私，建议**用温和、非诊断式**的题目，且放在问卷后段。

---

## 2. 自我特质 & 在关系中的表现

> 注：本部分只关注「和亲密关系高度相关」的特质，不做完整人格测验。

### 2.1 关系中的沟通与表达风格

- **平时表达方式**：  
  - 喜欢把想法直接说出来 vs 更含蓄委婉。  
  - 遇到不开心，是会主动说、写长消息、先冷静一下再说，还是倾向自己消化。  
- **倾听 vs 表达平衡**：  
  - 更擅长倾听 vs 更容易不自觉变成自己讲。  

> 标记：✅ 核心维度  
> 说明：现有问卷更多是描述「理想的对方」，这里补上**「我自己在关系里是怎么说话的」**。

---

### 2.2 冲突处理与修复方式

- 遇到矛盾时常见反应：  
  - 当场说清楚 / 先冷静后再说 / 选择回避 / 情绪爆发后再修复等。  
- 对「道歉」的态度：  
  - 能否主动道歉、如何表达歉意（语言、行动等）。  
- 面对对方情绪时的惯常做法：  
  - 先安抚 vs 先讲道理 vs 转移话题等。  

> 标记：✅ 核心维度  
> 说明：现有问卷已经有「你不适应对方的某些情绪方式」，但缺少**自己在冲突中的风格**这部分。

---

### 2.3 情绪调节风格（行为化）

- 遇到压力或心情不好时常用的方式：  
  - 找人倾诉 / 自己想办法解决 / 用娱乐或游戏转移注意力 / 运动 / 工作过度等。  
- 典型的情绪恢复时间：  
  - 当天就恢复 / 需要几天 / 往往会记很久。  

> 标记：✅ 核心维度  
> 说明：不出现「神经质、情绪调节能力」等心理学术语，用具体场景描述。

---

### 2.4 安全感 & 亲密距离偏好（行为版）

- 联系频率的舒适区：  
  - 每天多次聊天 / 每天简单聊聊 / 几天一次也没问题。  
- 对「对方有自己的社交圈」的感受：  
  - 感到安心 / 偶尔介意 / 比较不安，需要更多参与。  
- 对手机隐私、社交账号的态度：  
  - 一定要互看 / 可以共享但不强求 / 各自保留空间等。  

> 标记：✅ 核心维度  
> 说明：用于在不使用「依恋类型」术语的前提下，测出**安全感与亲密距离偏好**。

---

### 2.5 自我成长与调整意愿

- 是否愿意为一段关系学习新东西（沟通方式、情绪管理、时间分配等）。  
- 对「自我反思」的看法：  
  - 经常自省 / 偶尔 / 很少。  
- 是否接受「一起尝试咨询 /课程/阅读相关书籍」这种方式来解决问题。  

> 标记：✅ 核心维度  
> 说明：即使两人不完全匹配，只要**愿意调整与学习**，长期关系成功率会高很多。

---

### 2.6 完整人格大五（开放性、尽责性等）

> 标记：🟡 已覆盖 / 不需要在问卷中做「整套大五量表」  
> 说明：  
> - 在本项目中，大五更多用于**后台画像建模**（如标签“偏外向”“偏尽责”），  
> - 而不是面向用户呈现「你是高开放性」这类结果。  
> - 相关题目已通过「性格描述、生活方式、工作习惯」等间接覆盖，不再由 AI 直接出完整测验题。

---

## 3. 生活方式与节奏匹配（增补部分）

> 大部分生活方式相关内容（吸烟饮酒、运动、饮食、家务、宠物、兴趣爱好、消费习惯等）已在现有问卷充分覆盖。

### 3.1 作息节奏与空闲时间

- 典型作息：偏早睡 / 夜猫子 / 指定时间。  
- 工作日与周末的典型安排：更多宅在家 / 喜欢外出活动 / 经常加班。  
- 每周可用于相处和线下见面的时间大概有多少（平日/周末）。  

> 标记：✅ 核心维度  
> 说明：有助于判断**现实可相处时间和节奏是否匹配**，这部分通常**尚未系统覆盖**。

---

### 3.2 社交节奏与独处时间需求

- 每周大概有多少社交活动（聚会、饭局、线下活动等）。  
- 需要独处时间的程度：  
  - 每天都需要 / 偶尔需要 / 不太需要。  

> 标记：✅ 核心维度  
> 说明：与对方的「社交喜好」结合，可以判断双方**节奏是否容易互相拖累或互相滋养**。

---

### 3.3 已覆盖的生活方式维度（不再出题）

> 标记：🟡 已覆盖  
> 这些内容已在现有问卷中详细覆盖，AI 不要重复造题：

- 吸烟、饮酒习惯与频率。  
- 运动与健康习惯（是否运动、频率、类型）。  
- 做饭与饮食偏好（重辣/清淡、外卖/在家做、对营养和身材的关注等）。  
- 家务与整洁度要求。  
- 宠物态度与养宠经验。  
- 兴趣爱好（运动、游戏、阅读、艺术、旅行、自我成长等）。  
- 消费观与金钱观（性价比、物质 vs 体验、债务态度、是否超前消费等）。  

---

## 4. 价值观 & 人生方向（补齐“内在三观”）

### 4.1 亲密关系在生活中的优先级

- 在当前阶段，对「谈恋爱 / 找对象」的重视程度：  
  - 绝对优先 / 重要但不是唯一 / 顺其自然。  
- 愿意为感情做的让步程度：  
  - 调整作息 / 调整工作安排 / 为对方搬家等（具体行为选项）。  

> 标记：✅ 核心维度  
> 说明：补上「感情在整体人生中的权重」，配合已覆盖的「人生阶段与规划」更完整。

---

### 4.2 婚姻、生育与家庭观

- 对婚姻的态度：  
  - 必要 / 可以不结婚但有稳定关系 / 不确定 / 暂时不考虑。  
- 对是否要小孩的态度：  
  - 一定要 / 可以有可以没有 / 不想要 / 以后再说。  
- 理想的结婚/组建家庭时间范围（几年内、年限区间）。  

> 标记：✅ 核心维度  
> 说明：婚姻与孩子是长期匹配中最关键的分叉之一，需明确。

---

### 4.3 家庭角色与分工观（性别角色期待）

- 对家务分工的理想期待：  
  - 大致均分 / 谁擅长谁多做 / 传统分工（例如谁主内谁主外）等。  
- 对另一方在经济上的承担期待：  
  - AA / 一方略多 / 可接受一方阶段性承担大头等（不用具体金额）。  
- 对「谁更照顾父母/孩子」的预期（平衡型 / 一方更多等）。  

> 标记：✅ 核心维度  
> 说明：补齐「现实生活中的性别角色与分工观」，原已覆盖维度更多侧重**工作状态与收入**。

---

### 4.4 已覆盖的价值观/人生规划向维度

> 标记：🟡 已覆盖  

- 在对方人生状态与未来规划上的不匹配（如：不重视亲密关系、完全不考虑稳定关系或结婚、事业至上、长期异地等）。  
- 工作状态与职业形态的考量（体制内、创业、自由职业、过于不稳定等）。  
- 经济状况与金钱处理方式的顾虑（收入过低或极不稳定、超前消费、债务、金钱控制欲、重大开支观念差异等）。  

---

## 5. 对象需求 & 相处模式期待（补关键但不过度重复）

### 5.1 联系频率 & 相处方式期待

- 希望对方多长时间联系一次比较舒服。  
- 理想的相处方式比例：  
  - 在家一起做事 / 外出吃饭看电影 / 各自做事偶尔互动 / 共同参与某项长期活动（运动、游戏、兴趣等）。  

> 标记：✅ 核心维度  
> 说明：相比「对方兴趣/性格」这些已覆盖内容，这一块更偏**关系节奏与日常模式**。

---

### 5.2 对对方独立性 / 陪伴度的期待

- 希望对方在生活上有多独立：  
  - 情绪、经济、生活技能等层面分别可以问。  
- 对「各自保持一部分个人生活」的看法：  
  - 非常重要 / 适度即可 / 不太在意，只要在一起。  

> 标记：✅ 核心维度  

---

### 5.3 已覆盖的「理想对象特质」维度

> 标记：🟡 已覆盖  

- 性格特质与处事风格（外向/内向、温柔体贴、理性/感性、独立自律、有责任感、幽默、重视家庭、重视成长、善于沟通或倾听等）。  
- 情绪状态与情绪能力（情绪稳定、乐观、共情能力、在压力下的沟通方式、情绪恢复速度等）。  
- 人生阶段与关系规划（是否在积极寻找关系、结婚时间预期、是否愿意为感情/工作换城市等）。  
- 工作状态与收入水平。  
- 对方的恋爱经历与经验。  

---

## 6. 需求–供给自评 & 潜在矛盾（本项目要重点新增）

> 这一块是前面规则中特别提到的「第四步：挖掘需求–供给不匹配与潜在矛盾」，需要单独建维度。

### 6.1 我能稳定提供的东西（自我供给）

- 在情绪方面：  
  - 我能给对方的更多是：情绪安抚 / 理性分析 / 陪伴倾听 / 一起行动解决问题。  
- 在时间方面：  
  - 每周可以固定拿出来的时间段与总时长。  
- 在经济方面：  
  - 对关系中经济支持/共同规划的态度（例如“可以承担共同生活的一半/更多/阶段性较少”等，不涉及具体金额）。  
- 在生活技能方面：  
  - 比如：做饭、家务、计划旅行、处理突发事件等。  

> 标记：✅ 核心维度  

---

### 6.2 我希望对方能提供的东西（需求）

- 在情绪方面的期待：  
  - 希望对方在我难受时更偏**安慰型**、**共情型**、**讲道理型**，还是**给实际帮助型**。  
- 在时间方面的期待：  
  - 希望对方能提供多少**稳定、高质量陪伴时间**。  
- 在经济与责任方面的期待：  
  - 在可接受范围内，希望对方的经济与责任承担大概在什么水平（描述成区段与态度）。  
- 在生活技能与生活组织方面：  
  - 更希望对方负责哪些事情（例如：主要做饭 / 主要做计划 / 主要管家务等）。  

> 标记：✅ 核心维度  

---

### 6.3 自觉的「可能不匹配点」

- 用户自己认为：  
  - **我想要的** vs **我目前能提供的**，在哪些方面有差距（例如时间、经济、情绪稳定度等）。  
- 用户对「如何面对这些不匹配」的态度：  
  - 期待对方包容 / 自己愿意调整 / 希望双方一起想办法。  

> 标记：✅ 核心维度  

---

## 7. 生成问卷题目时必须注意的要点（供 AI 使用）

这一节是给**出题引擎**看的，不是给用户看的。

### 7.1 先分阶段，再出题（整体路径）

- **第一步：背景与基本情况**  
  - 基本人群信息 + 核心生活状态（城市、职业、作息、家庭责任）。  
- **第二步：自我特质与生活方式**  
  - 只保留与关系高度相关的部分（沟通方式、冲突处理、节奏、生活习惯）。  
- **第三步：价值观与人生方向**  
  - 婚姻、生育、家庭分工、亲密关系优先级。  
- **第四步：对“对方”的需求与期待**  
  - 不再重复已覆盖维度，而是重点问**节奏、相处模式、独立性期待**等新增部分。  
- **第五步：需求–供给差距 & 自我觉察**  
  - 用户认为自己能提供什么、想要什么、哪里可能不匹配，以及面对不匹配的态度。  

---

### 7.2 一定要做的几件事

1. **坚持“系列/分支问题”思路**  
   - 前一道题确定方向，后一道题往里挖。  
   - 例如：  
     - Q1：你希望对方多久联系你一次？  
       - 如果选「每天都聊」→ Q2 深挖「不联系多久会不安」。  
       - 如果选「几天一次也可以」→ Q2 问「你更在意见面质量还是频率」。  

2. **选项必须具体，禁止「其他」**  
   - 所有选项都要是具体、可区分的行为或状态。  
   - 可以加「很难说/看情况」这样的选项，**但不要写“其他”**。  

3. **避免心理学术语，尤其是对用户直呼**  
   - 不要问「你属于哪种依恋类型？」  
   - 要问的是：在具体场景下你会怎么做、怎么想。  
   - 如：对方 3 小时没回复消息，你更可能：继续忙自己的事 / 开始担心 / 生气 / 主动再发一次等。  

4. **默认用户是单身，注意措辞**  
   - 用「对方」「未来的那个人」「你要找的人」，  
   - 不用「伴侣」「你们现在的关系」之类容易让人误解成已在恋爱中。  

5. **可以适度问“过去的恋爱经历”，但不要过度挖隐私**  
   - 问「在过去的关系中，你最介意哪类问题」，  
   - 不要问具体人名、事件细节。  

---

### 7.3 要避免的几个坑

1. **避免巴纳姆效应（谁看都像说自己）**  
   - 问题和反馈都要尽量具体、带有区分度，不能是「你有时需要安全感，有时又渴望自由」这种废话。  

2. **避免「无意义/模糊/过宽泛」题目**  
   - 不要问「你喜欢什么样的生活？」这种太大太空的问题。  
   - 要拆成「工作强度」「闲暇活动」「物质 vs 体验」「是否愿意搬家」等具体维度。  

3. **避免强引导和“道德绑架”**  
   - 不要问「你是不是觉得有钱的对方更可靠？」  
   - 应改成中性表述：“在选择对方时，你会多大程度考虑对方的经济稳定性？”  

4. **避免重复问同一个维度**  
   - 所有在「🟡 已覆盖维度」里的内容，**AI 不要再造题**，除非是为了分支路由（如「你是否愿意进一步回答关于XX的问题」）。  

5. **避免过度负面/刺激性的提问**  
   - 不直接问「你是否因为经济问题被分手过？」  
   - 可以问「在金钱相关的问题上，你觉得过去的自己有哪些遗憾或经验教训？」（仍然要谨慎）。  

---

### 7.4 题目形式与语言风格

- 尽量采用**行为化 + 情境化**描述，如：  
  - “如果约好的见面时间对方迟到 30 分钟且没发消息，你通常会怎么做？”  
- 优先使用**7 点量表**或多选题，但保持口语化：  
  - 「完全不在意 → 有点在意 → 非常在意」  
- 保持语气中性、尊重，不给用户贴标签。  
- 可以适度加入「自我觉察」类开放题，鼓励用户反思，增强「认识自己」这一产品价值。

---

> 本 v2 指南的目标：  
> - 帮你过滤掉对相亲场景不重要或已被覆盖的维度，  
> - 补足「节奏匹配」「关系风格」「需求–供给差距」这几块关键内容，  
> - 同时把问卷生成规则（分支、选项、措辞）写清楚，方便后续用 AI 稳定出题。

## JSONL 问卷文件格式规范

> 这是一份「通用的 JSONL 问卷格式规范」，**不依赖具体题号、具体 section 名称、也不依赖任何已有文件名**。  
> 你可以用它来设计全新的问卷，或转换已有问卷到 JSONL 形式。

---

## 一、总体结构

- 文件采用 **JSONL（JSON Lines）** 格式：  
  - 每一行是一个完整的 JSON 对象，代表一题。  
  - 行与行之间用换行分隔，不需要数组包裹、行末不加逗号。  

- 每个 JSON 对象表示一题，统一称为「question 对象」。  

---

## 二、字段一览（必须字段）

每个 question 对象建议只使用下面这些字段，**所有都是必填字段**：

| 字段名        | 类型              | 说明                                                     |
|---------------|-------------------|----------------------------------------------------------|
| id          | string            | 题目唯一编号，如 "Q1", "Q2"，在一份问卷内不重复。    |
| section     | string            | 所属部分的名称（任意字符串），用于把题目分块组织。      |
| section_id  | integer           | 所属部分的编号（任意正整数），只要求在同一问卷内自洽。  |
| topic       | string            | 题目主题/标题，简短概括这道题在问什么。                 |
| content     | string            | 题干全文，用户真正看到并回答的那句话/那段话。           |
| type        | string            | 题型："single_choice" / "multiple_choice" / "open_ended"。 |
| options     | array\<object\>   | 选项数组；开放题为 []。                               |
| dimension   | array\<string\>   | 此题测量的维度标签列表，用于分析和分组。                 |
| dependency  | object            | 依赖条件（什么时候需要出这道题）。                       |
| note        | string            | 备注，给设计者或系统看的短说明（可以是空字符串）。       |

> 约定：  
> - **不再使用** modification、Q新0.1 等历史字段和命名。  
> - 如果将来需要额外元数据，可以另起字段，但要保证现有字段含义不变。  

---

## 三、各字段含义与简单示例

### 3.1 id

- 唯一标识一题，如 "Q1", "Q15", "Q101"。  
- 在一份问卷内部不能重复，是否连续不做强制要求。  

**示例：**

json
"id": "Q15"


---

### 3.2 section 与 section_id

- section：人类可读的部分名称，如 "金钱与消费", "信任与亲密"，也可以是 "Part A"。  
- section_id：对应的数字编号，如 1, 2, 3。  

规范只要求：  
- 同一份问卷里，**同一个 section 名称始终对应同一个 section_id**；  
- 不强制有多少个 section，也不限制具体的名称和编号。

**示例：**

json
"section": "金钱与消费",
"section_id": 1


---

### 3.3 topic

- 简短的题目主题，用来快速知道「这一题大概问什么」。  
- 不要求像题干那样完整，只要能概括核心即可。

**可以的写法：**

json
"topic": "个人消费态度"


**不推荐的写法：**

- 写成一个很长的完整句子（会和 content 混在一起）。  
- 包含题号、依赖条件等技术信息。

---

### 3.4 content

- 用户真正看到并回答的题干文本。  
- 只描述**情境 / 感受 / 行为**，不写技术性的说明。  

**必须避免的内容：**

- 不要写依赖条件：  
  - ❌ 如果你在 Q3 选了 A 或 B，这道题才适用……  
- 不要写类型说明：  
  - ❌ （多选）、（开放式） 等；这些由 type 决定。  

**示例：**

json
"content": "你对消费的总体态度是什么？"


---

### 3.5 type

- 可选值只有三种：

json
"type": "single_choice"    // 单选题
"type": "multiple_choice"  // 多选题
"type": "open_ended"       // 开放式/简答题


---

### 3.6 options

- 对于单选、多选题：是一个选项对象数组。  
- 对于开放题：必须是 []（空数组）。

**单选题示例：**

json
"options": [
  {"key": "A", "text": "消费是生活享受，应该尽情花费"},
  {"key": "B", "text": "消费主要用于满足需求和一些享受"},
  {"key": "C", "text": "消费应该谨慎，但必要时可以花费"},
  {"key": "D", "text": "消费应该极其谨慎和有限"},
  {"key": "E", "text": "消费越少越好，应该最小化花费"}
]


**开放题示例：**

json
"options": []


---

### 3.7 dimension

- 用来标记「这道题在测什么」，一般是 1–3 个简短标签。  
- 这些标签在不同题目之间可以复用，用于后续打分、出报告。

**示例：**

json
"dimension": ["个人消费价值观", "消费风格"]


---

### 3.8 note

- 给设计者或系统的简短备注，不参与评分逻辑。  
- 常见用法：  
  - "冲突检测"、"背景题"、"系列追问"、"总结题" 等。  
- 不需要时可以为 ""。

**示例：**

json
"note": "冲突检测"


---

## 四、dependency 字段（依赖逻辑）

dependency 用来描述：**在什么条件下需要出这一题**。  

### 4.1 通用结构

json
"dependency": {
  "type": "none"  // 或其它类型
  // 其他字段根据 type 决定
}


本规范中推荐保留的几种简单类型如下（够用又容易实现）：  

1. "none"：无依赖  
2. "any_answer"：只要某题答过即可  
3. "option_match"：指定题目 + 指定选项  
4. "multi_option"：多个条件同时满足（AND）  
5. "background"：依赖自然语言描述的背景条件  

> 说明：历史上还存在更复杂的类型（如 range_composite, multi_composite, similar 等），如果后续确实需要，可以在此基础上继续扩展；本规范先给出一套**最小可用子集**。

---

### 4.2 type: "none"

**结构：**

json
"dependency": {
  "type": "none"
}


**含义：**

- 不依赖任何其它题目或属性，这是一道所有人都会看到的基础题。  

---

### 4.3 type: "any_answer"

**结构：**

json
"dependency": {
  "type": "any_answer",
  "ref": "Q7"
}


**含义：**

- 只要用户回答过 id = "Q7" 的那道题（选了任意选项），这道题就会出现。  
- 常用于「系列追问」：先问一个总问题，再根据是否回答过展开细节。  

---

### 4.4 type: "option_match"

**结构：**

json
"dependency": {
  "type": "option_match",
  "ref": "Q3",
  "options": ["D", "E"]
}


**含义：**

- 当用户在 Q3 中选择了 D 或 E 时，这道题才会出现。  
- 用于「如果你经常这样/问题比较严重，那么再多问一题」。  

---

### 4.5 type: "multi_option"（多条件 AND）

**结构：**

json
"dependency": {
  "type": "multi_option",
  "operator": "AND",
  "conditions": [
    {"ref": "Q1", "options": ["A", "B"]},
    {"ref": "Q2", "options": ["D", "E"]}
  ]
}


**含义：**

- 上面例子表示：  
  - 用户在 Q1 选了 A 或 B，**并且** 在 Q2 选了 D 或 E，  
  - 两个条件都满足，这道题才出现。  
- 典型用法：**冲突检测、矛盾识别**。  

---

### 4.6 type: "background"（自然语言背景条件）

历史版本里，背景依赖是通过引用「某道背景题」来实现的。  
在本规范中，背景依赖统一改为**自然语言描述**，不再要求一定对应某个选择题。

**结构：**

json
"dependency": {
  "type": "background",
  "BackgroundContent": "只有当用户正在评估某一个具体的人（而不是纯理想对象）时，这道题才需要回答。"
}


**规则与说明：**

- BackgroundContent 是一个任意字符串，用自然语言清楚说明：  
  - 想要出这道题，**用户需要具备怎样的背景 / 情境 / 身份**。  
  - 例如：  
    - 「只有有过亲密关系经验的人才回答」  
    - 「只有当用户正在回顾前任时才需要回答」  
- 解析器可以先用属性/题目推断，如果推断不出来，也可以直接把这句文本展示给其他系统或人工判断。  
- 这样适合表达「很难用简单选项逻辑写清楚」的复杂前提。

## 六、简单校验清单

在生成或检查 JSONL 文件时，可以按下面清单做快速验证：

1. 每一行是否是合法 JSON 对象？  
2. 是否都包含这几个字段：  
   - id, section, section_id, topic, content, type, options, dimension, dependency, note。  
3. id 是否在整个文件内不重复？  
4. type 是否只使用了三种合法值之一？  
5. 开放题的 options 是否为 []？  
6. dimension 是否为字符串数组？  
7. dependency.type 是否属于本规范支持的几种类型之一？  
8. 使用 option_match 或 multi_option 时，是否正确写了 ref 和 options？  
9. 使用 background 时，是否填写了清晰的 BackgroundContent 文本？  

只要满足以上规则，这份 JSONL 问卷就可以被后续系统稳定地解析、跳题和生成高质量对话。
